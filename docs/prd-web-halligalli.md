# Halligalli 网页端练习应用 PRD

## 1. 文档信息

- 产品名称：Halligalli 网页端练习应用
- 文档版本：v1.0
- 文档状态：草案
- 创建日期：2026-03-12
- 目标：定义一个面向网页端的 Halligalli 练习产品，用于帮助用户熟悉规则、训练反应速度、提升识别与拍铃准确率。

## 2. 产品背景

Halligalli 是一个强调观察、计算与快速反应的桌游。在线下体验中，玩家需要快速判断桌面水果牌上同类水果数量之和是否等于 5，并在条件满足时第一时间拍铃。对于初学者和希望提升反应能力的玩家来说，存在以下问题：

- 线下开局成本较高，需要实体牌和多人参与。
- 新手不熟悉“同类水果相加等于 5 才拍铃”的核心规则，容易误判。
- 玩家难以独立进行高频练习，也无法量化自己的反应速度和正确率。
- 没有系统化的训练模式帮助用户从规则理解过渡到高强度实战。

因此，需要一个网页端练习应用，降低学习门槛，支持随时打开即练，并通过数据反馈帮助用户持续提升。

## 3. 产品目标

### 3.1 业务目标

- 在网页端提供低门槛、即开即用的 Halligalli 练习体验。
- 让新用户在 3 分钟内理解基本规则并完成首次练习。
- 建立可复用的训练体系，为后续排行榜、社交对战和付费内容打基础。

### 3.2 用户目标

- 快速理解 Halligalli 判定规则。
- 通过单人练习提升识别与拍铃反应速度。
- 在训练后查看成绩、错误原因和进步趋势。

### 3.3 成功指标

- 新用户首次进入后完成一局练习的比例 >= 70%
- 教学完成率 >= 60%
- 练习模式平均局时长 >= 2 分钟
- 用户平均正确率 >= 75%
- 用户 7 日内二次访问率 >= 25%

## 4. 用户画像

### 4.1 核心用户

1. Halligalli 新手
   - 想快速学会规则
   - 容易混淆“单张牌数量”和“场上同类总和”

2. 桌游玩家
   - 希望在聚会前练手
   - 关注速度、判断稳定性和成绩对比

3. 反应训练用户
   - 对快反类小游戏有兴趣
   - 更关注分数、连击、反应时和个人成长

### 4.2 使用场景

- 第一次接触 Halligalli，想通过网页理解玩法
- 聚会前临时练习，找回判断节奏
- 碎片化时间里进行 1 到 3 分钟的快速训练
- 连续多局训练，挑战更高难度和更快节奏

## 5. 产品定位

Halligalli 网页端练习应用是一款以“规则学习 + 单人训练 + 数据反馈”为核心的轻量网页产品。产品初期聚焦单人练习体验，不做复杂社交和多人实时对战，优先验证用户是否愿意持续训练及其最关注的反馈指标。

## 6. 范围定义

### 6.1 本期范围（MVP）

- Web 端适配桌面浏览器与移动浏览器
- Halligalli 基础规则教学
- 单人练习模式
- 拍铃判定与反馈
- 分数、正确率、反应时统计
- 训练结算页
- 基础设置项（音效、速度、难度）

### 6.2 后续范围（非 MVP）

- 用户登录与历史战绩同步
- 排行榜
- 每日挑战
- 好友对战 / 实时多人模式
- 成就系统
- AI 陪练或自适应难度

## 7. 核心问题与解决方案

### 7.1 问题 1：用户不理解规则

解决方案：

- 提供 3 步内完成的交互式新手教学
- 每次误判后明确提示“为什么错”
- 在首页用简洁语言强调唯一核心规则：同类水果总数等于 5 才能拍铃

### 7.2 问题 2：用户难以持续练习

解决方案：

- 将单局时间控制在 1 到 3 分钟
- 提供难度梯度和重新开始入口
- 通过结算页展示成绩与提升建议

### 7.3 问题 3：用户无法感知成长

解决方案：

- 展示平均反应时、最快反应、正确率、误拍次数
- 对比本局与个人历史最佳成绩
- 为不同错误类型做分类统计

## 8. 核心玩法定义

## 8.1 Halligalli 基础规则

- 牌面包含多种水果，每张牌展示 1 到 5 个水果中的某个数量
- 桌面上会逐步出现新的水果牌
- 当当前桌面上“同一种水果”的总数恰好等于 5 时，用户应立即拍铃
- 若用户在可拍时拍铃，判定为正确
- 若用户在不可拍时拍铃，判定为错误

## 8.2 网页端练习抽象

为适配单人练习，系统模拟“公共桌面持续翻牌”的过程：

- 屏幕中央持续出现多列水果堆，每次翻出新牌
- 用户需要观察所有可见牌面，判断是否存在某种水果总数为 5
- 一旦满足条件，用户点击拍铃按钮或按快捷键触发拍铃
- 系统即时反馈正确或错误，并推进下一轮

## 8.3 判定原则

- 以“当前可见牌面”中同类水果的总和为准
- 同时若存在多个满足条件的水果种类，只要拍铃即算正确
- 如果在规定反应窗口内未拍铃，则记为漏拍
- 错拍、漏拍均影响分数和统计指标

## 9. 功能需求

## 9.1 首页

### 目标

帮助用户快速理解产品并进入练习。

### 功能点

- 产品标题与一句话说明
- “开始练习”主按钮
- “新手教学”次按钮
- 当前模式说明
- 最近成绩卡片（如未登录则基于本地缓存）
- 基础设置入口

### 成功标准

- 用户进入首页后 5 秒内能理解这是一个 Halligalli 训练产品
- 用户能在 1 次点击内开始第一局

## 9.2 新手教学

### 目标

让用户在短时间内理解拍铃条件与常见错误。

### 功能点

- 分步骤讲解规则
- 静态示例：哪些情况可以拍铃，哪些不可以
- 动态演示：系统翻牌，提示用户是否该拍
- 教学结尾小测试（3 到 5 题）

### 教学内容建议

1. 认识水果牌与数量
2. 理解“同类水果总数 = 5”
3. 学会区分“刚好等于 5”与“大于 5”
4. 练习拍铃时机

### 通过条件

- 小测试正确率达到 80% 以上，或允许跳过进入练习模式

## 9.3 单人练习模式

### 目标

提供连续、高频、低等待的核心训练体验。

### 核心机制

- 系统按节奏翻牌
- 用户观察当前桌面
- 用户在判断满足条件时拍铃
- 系统即时判定并累计成绩

### 可配置项

- 难度：简单 / 标准 / 困难
- 节奏速度：慢 / 中 / 快
- 局长：60 秒 / 120 秒 / 无尽
- 输入方式：点击拍铃 / 空格键拍铃（桌面端）
- 音效开关

### 难度建议

- 简单：水果种类少、翻牌速度慢、可视牌面少
- 标准：标准水果池与正常节奏
- 困难：翻牌更快、干扰更多、判定窗口更短

## 9.4 拍铃反馈系统

### 目标

保证用户明确知道自己为何得分或失分。

### 正确反馈

- 铃声 + 成功动效
- 高亮满足条件的水果组合
- 显示本次反应时
- 增加分数 / 连击

### 错误反馈

- 错误音效 + 红色提示
- 明确指出错误原因：
  - 当前没有任何水果总和等于 5
  - 用户误把大于 5 当成等于 5
  - 用户忽略了水果种类必须相同

### 漏拍反馈

- 当系统检测到应拍未拍时，给出轻提示
- 记录为漏拍，不打断整局节奏或仅轻度打断

## 9.5 结算页

### 目标

帮助用户复盘并驱动下一局训练。

### 展示内容

- 总得分
- 正确拍铃次数
- 错拍次数
- 漏拍次数
- 正确率
- 平均反应时
- 最快反应时
- 连击记录
- 与历史最佳对比

### 行动入口

- 再来一局
- 调整难度
- 查看教学
- 返回首页

## 9.6 本地成绩记录

### 目标

在无登录前提下保留基础成长记录。

### 功能点

- 使用浏览器本地存储保存最近成绩
- 保存历史最佳成绩
- 保存常用设置项
- 可在首页读取并展示摘要

## 11.1 新用户首次使用流程

1. 进入首页
2. 点击“新手教学”
3. 完成规则说明与测试

## 12. 交互与体验要求

## 12.1 视觉原则

- 风格轻松、有节奏感、具备桌游氛围
- 重点信息突出：水果数量、拍铃按钮、对错反馈
- 避免界面信息过载，优先保障判定可读性

## 12.2 交互原则

- 拍铃入口始终固定在易触达区域
- 成功与失败反馈必须在 300ms 内可感知
- 所有关键操作尽量单击完成
- 桌面端支持键盘快捷操作，移动端优先按钮点击

## 12.3 响应式要求

- 桌面端：适合键盘与鼠标操作，拍铃按钮明显
- 移动端：单手可点击主要操作区，核心信息置中
- 横屏优先优化练习页，竖屏需可正常使用

## 13. 规则与系统设计要求

## 13.1 牌组设计

- 水果种类建议：4 到 5 类
- 每种水果有不同数量的牌
- 系统需支持随机发牌，但要保证训练节奏合理

## 13.2 回合节奏

- 系统每隔固定时间翻一张牌或一组牌
- 每次翻牌后实时计算当前是否存在可拍条件
- 若触发可拍条件，开始计时反应窗口

## 13.3 计分建议

- 正确拍铃：+100 分
- 连击奖励：连续正确可获得额外加分
- 错拍：-50 分
- 漏拍：-30 分
- 反应越快，奖励越高（可加速度分）

## 13.4 错误分类

- 错拍：不该拍时拍了
- 漏拍：该拍时没拍
- 慢拍：拍了但超过有效窗口

### 关键埋点事件

- 教学完成
- 教学跳过
- 练习开始
- 正确拍铃
- 错误拍铃
- 漏拍发生
- 练习结束
- 再来一局点击
- 设置项变更

### 关键属性

- 设备类型
- 难度等级
- 速度等级
- 局时设置
- 本局分数
- 正确率
- 平均反应时

## 15. 非功能需求

## 15.1 性能要求

- 首页首屏加载时间 < 2 秒
- 练习中核心交互响应 < 100ms
- 动效与翻牌过程流畅，无明显掉帧

## 15.2 可用性要求

- 无登录即可开始使用
- 主要按钮清晰可见
- 新用户在不阅读长文本的情况下也能快速上手

## 15.3 兼容性要求

- 支持主流现代浏览器
- 支持移动端 Safari / Chrome
- 适配常见手机与桌面屏幕尺寸

## 15.4 可访问性要求

- 关键按钮具备足够对比度
- 支持键盘触发主要操作
- 颜色反馈之外辅以文字或图标反馈

## 16. 风险与待确认项

### 16.1 产品风险

- 如果练习过于机械，用户可能快速流失
- 若视觉呈现不清晰，会直接影响判定体验
- 若反馈太弱，用户难以建立正确规则理解

### 16.2 实现风险

- 实时判定与动画节奏需要严格同步
- 移动端点击延迟和误触需重点优化
- 不同难度下的随机牌面需要平衡，避免过难或过易

### 16.3 待确认项

- 是否需要完全复刻原版牌组，还是允许训练化改造
- 是否需要账号体系
- 是否需要日榜或好友排名
- 是否需要音效、震动等更强反馈
- 是否支持键盘之外的更多输入方式

## 17. MVP 验收标准

- 用户可在首页直接开始一局练习
- 用户可完成一次基础教学并理解规则
- 系统可正确判定正确拍铃、错拍、漏拍
- 练习结束后可展示完整成绩统计
- 支持桌面与移动端基本可用
- 本地可保存最近成绩与设置

## 18. 里程碑建议

### Phase 1：产品定义与原型

- 完成 PRD
- 完成低保真流程图与页面原型
- 确认核心规则与 MVP 边界

### Phase 2：MVP 开发

- 搭建前端项目
- 完成首页、教学、练习、结算主流程
- 完成本地存储与基础埋点

### Phase 3：测试与优化

- 验证不同设备体验
- 调整难度与计分平衡
- 优化反馈、性能与留存体验

## 19. 后续设计与开发建议

- 先以“单人训练”做深，再扩展“挑战”和“对战”
- 教学与练习应共享同一套判定引擎，避免规则不一致
- 练习页应优先围绕“看得清、反应快、反馈准”设计
- 数据结构从一开始就预留成绩记录和扩展模式字段
