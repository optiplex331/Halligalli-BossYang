# v0.6.0 Behavior Baseline

This is the migration characterization for the standalone Halligalli release. It
captures observable behavior and explicitly identifies the V1 decisions that
replace it. It is not a request to preserve the Node.js, Socket.IO, React state,
timer, CSS-selector, or `localStorage` implementation.

## Observable behavior matrix

| Surface | v0.6.0 observable contract | Later fixture or journey seam |
| --- | --- | --- |
| Card inventory and table | Four fruits in canonical order: `banana`, `strawberry`, `lemon`, `grape`; 72 cards total; for each fruit, 1/2/3/4/5 appears 3/5/5/3/2 times. The public product supports tables of 3–6 seats, deals round-robin, and only a seat's top face-up card contributes to totals. | Data fixture for catalog/deal/totals; preserve the canonical first-fruit choice if more than one total is five. |
| Bell Window and card movement | A window is open iff a visible fruit total is exactly five. A correct ring collects all face-up cards, clears the table, and gives the winner the next turn. An incorrect ring pays `ceil(face-up table cards / 2)`, drawing before recycled won cards and placing paid cards beneath the existing face-up pile. An unhandled window becomes a miss before the next flip and once at finalization. | Transition fixture for correct, wrong, missed, and pending-final-window cases. |
| Single-Player Path | Settings expose 3/4/5/6 players; easy/normal/hard reveal intervals are 1850/1400/900 ms; score-bonus windows are 1900/1500/1000 ms; duration choices are 45/60/90 seconds. A 3–2–1 countdown precedes an immediate first flip; the bell button and Space invoke the same action while play is active. | Browser journey through setup, countdown, Space/button bell, result, and replay; timing assertions use fake clocks rather than wall time. |
| Scoring and summary | Correct score is `120 + 6 * collected + max(0, Math.round((bonusWindow - reactionMs) / 20)) + 10 * priorStreak`. Wrong rings reset streak and assess 50 plus 4 per card actually paid; a missed window assesses 30. Accuracy is `correct / (correct + wrong + missed)` or zero; average reaction time uses JavaScript `Math.round`; best reaction is the minimum. | Language-neutral score vectors must carry integer expectations around JavaScript half-rounding behavior. See the intentional V1 score-ledger correction below. |
| Multiplayer Authority | The server owns shuffle/deal, first flip, later flips, clocks, Bell Window, reaction time, penalty, score, result, and end state. Clients send room intents and a payload-free bell intent; they do not supply score, winner, cards, or reaction time. Two competing bell intents are processed serially: after a valid first result, a later one is wrong. | Authority transcript fixture and black-box WebSocket journey; do not freeze Socket.IO event names or payload envelopes. |
| Rooms | Public creation and start accept only 3–6 participants. A four-character uppercase code excludes `I`, `O`, `0`, and `1`; joins normalize with trim and uppercase. Only the host starts after all current players are ready. Empty lobbies disappear; active rooms become eligible for dissolution after 10 minutes of inactivity; all-disconnected active rooms are destroyed. There is no durable room or player recovery. | Room lifecycle transcript with create/join/ready/start/error/expiry facts; protocol is free to move from Socket.IO to native WebSocket. |
| Bilingual, accessibility, and mobile | Stable copy is present in `COPY.zh` and `COPY.en`. The baseline has focused screen regions, status/live announcements, keyboard Space bell input, an accessible bell name, responsive table breakpoints, 44 px mobile control minimums, and reduced-motion rules. | Role/name-based browser journeys in both languages at 320/390/520 px plus reduced-motion emulation. Preserve usability, not DOM/CSS structure. |

Primary baseline sources are the pinned [catalog and game rules](https://github.com/optiplex331/Halligalli-BossYang/tree/916c65a24c782d09783e1881f0cbf5ddba92ef6c/src/game), [single-player shell](https://github.com/optiplex331/Halligalli-BossYang/blob/916c65a24c782d09783e1881f0cbf5ddba92ef6c/src/App.tsx), [room model](https://github.com/optiplex331/Halligalli-BossYang/blob/916c65a24c782d09783e1881f0cbf5ddba92ef6c/server/Room.ts), and [authoritative engine](https://github.com/optiplex331/Halligalli-BossYang/blob/916c65a24c782d09783e1881f0cbf5ddba92ef6c/server/GameEngine.ts).

## Storage surface

The release source uses browser `localStorage`; no product calls to
`sessionStorage`, IndexedDB, or browser cookies were found in the materialized
tree. The complete Halligalli key surface is:

| Key | v0.6.0 content | V1 handling |
| --- | --- | --- |
| `halligalli_settings` | Normalized `{ difficulty, duration, playerCount, language, soundEnabled }`; defaults are `normal`, `60`, `4`, `en`, and `true`. Baseline validation accepts player counts 3–6. | Retain as the sole Local Preferences record. Extend its accepted player count for the new two-seat path without adding progress data. |
| `halligalli_best` | Best round summary. | Remove during migration; never recreate. |
| `halligalli_recent` | Most recent round summary. | Remove during migration; never recreate. |
| `halligalli_history` | Up to 100 persisted round entries, including mode and timestamp. | Remove during migration; never recreate. |
| `halligalli_daily` | Day-scoped completed-round and goal state. | Remove during migration; never recreate. |
| `halligalli_achievements` | Achievement timestamps. | Remove during migration; never recreate. |

The [persistence implementation](https://github.com/optiplex331/Halligalli-BossYang/blob/916c65a24c782d09783e1881f0cbf5ddba92ef6c/src/game/persistence.ts) and [persistence tests](https://github.com/optiplex331/Halligalli-BossYang/blob/916c65a24c782d09783e1881f0cbf5ddba92ef6c/src/__tests__/persistence.test.ts) are the evidence for this table. A storage-removal contract deliberately fails against this release: seed all six keys, migrate, assert that only normalized settings remain, and assert that an active or completed match cannot recreate the other five.

## Explicit V1 boundaries

These are intentional changes, not baseline regressions.

| Boundary | Baseline fact | V1 contract |
| --- | --- | --- |
| **Two-seat multiplayer** | The public create and start paths reject two players with bilingual “3 to 6” errors. `Room.allReady()` having a generic two-player minimum is not shipping two-player support; [the server integration test](https://github.com/optiplex331/Halligalli-BossYang/blob/916c65a24c782d09783e1881f0cbf5ddba92ef6c/server/__tests__/room-size.test.ts) proves the public rejection. | **New V1 behavior:** two participants may create, enter, ready, start, play, finish, and continue under the same rules—no two-player rule variant. Historical rejection is evidence only, never a V1 conformance assertion. |
| Durable gameplay state | The five progress keys above are actively written after a solo round. | Delete them and retain only `halligalli_settings`; active round/result data stays in memory. |
| Score ledger | Live score floors after baseline penalties, while the baseline breakdown records nominal penalties and final summary derives from that breakdown. A penalty at zero can therefore create hidden debt against a later award. | Apply the Transition Score Floor on every authoritative transition and record only Applied Penalties, allocating a wrong-ring deduction to base penalty before card penalty. The breakdown becomes the sole score ledger. |
| Room continuity and seats | Lobby leave compacts seat indices; the standalone runtime does not provide V1's post-match continuation model. | A six-slot room keeps stable Seat Indices; its lifecycle is `lobby -> playing -> post_match -> lobby`, and two or more continuing participants may start another match. |
| Configuration and accessibility cleanup | The room projection carries language, the client can derive layout from local settings, `index.html` is fixed to `lang="en"`, and the Room Code input has no explicit label. | Authority owns room participant count/difficulty/duration; language and sound remain Local Preferences. Selected locale updates document language and Room Code has an explicit accessible label. |

## Minimal fixture and browser-journey handoff

The following are deliberately small, concrete vectors for the later data-only
Shared Behavior Contract. They are enough to catch semantic drift without
recreating the old TypeScript implementation.

1. `catalog_inventory`: assert 72 cards and the exact per-fruit distribution.
2. `visible_totals_first_match`: top cards banana 2 + banana 3 + grape 5 yield
   a Bell Window for `banana`, because canonical fruit order wins the tie.
3. `correct_bell_easy_250ms`: one visible banana 5, one collected card, easy
   mode, zero streak, and 250 ms reaction yields `correctHits = 1`, score 209,
   and breakdown additions 120/6/83/0 for base/collection/speed/streak.
4. `wrong_bell_payment`: use a deterministic pile to assert ceiling-half table
   payment, draw-then-won recycling, actual-card-count pricing, and an updated
   Bell Window after payment.
5. `pending_window_finalization`: an unhandled final grape window produces one
   missed hit and the 30-point missed transition exactly once.
6. `room_authority_3_to_6`: a non-host or unready room cannot start; an engine
   emits the first flip, measures server reaction time, and makes a second bell
   after a handled window wrong. Add a separate V1-positive two-seat transcript.
7. `storage_subtraction`: seed every baseline key and verify only normalized
   settings survive migration. Keep this separately from baseline conformance
   because it intentionally fails on v0.6.0.

Browser journeys should use roles and accessible names, not CSS selectors. One
single-player journey should cover Chinese and English setup, countdown, Space
and button bell activation, a live announcement, result view, and replay at the
three reference viewports. Multiplayer journeys should cover a representative
three-seat and six-seat path, then add the V1-only two-seat path. Reduced-motion
is a positive behavior check, not a screenshot comparison.

In the materialized release tree, using Node `v24.18.0` and pnpm `11.0.9`:

- `pnpm exec vitest run src/__tests__/rules.test.ts server/__tests__/GameEngine.test.ts server/__tests__/room-size.test.ts` passed: 3 files, 12 tests.
- `pnpm run test` passed earlier in the same materialized tree: 10 files, 66 tests.

The first command is the intentionally small ongoing characterization check.
The full suite and typecheck establish that the source material used for this
contract was internally consistent; later migration work should create V1
fixtures and browser journeys rather than adding more tests to this historical
standalone tree.
